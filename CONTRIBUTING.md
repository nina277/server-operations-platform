# コントリビューションガイド

## ブランチ戦略: GitHub Flow

本リポジトリでは **GitHub Flow** をブランチ戦略として採用する。`develop` のような長期運用ブランチは置かず、`main` を唯一のトランクブランチとする。

### 基本ルール

1. `main` は常にデプロイ可能な状態を保つ。壊れたコードを直接コミットしない。
2. 作業を始めるときは、必ず `main` から最新を取得してブランチを作成する。

       git checkout main
       git pull origin main
       git checkout -b feature/xxx

3. ブランチ名は変更内容が分かる名前にし、以下のプレフィックスを使う。
   - `feature/<説明>` : 新機能・新チケット実装
   - `fix/<説明>` : バグ修正
   - `chore/<説明>` : 雑務・設定・依存関係更新
   - `docs/<説明>` : ドキュメントのみの変更
4. こまめにコミットし、ブランチを origin へ push する。
5. 作業がある程度まとまった時点、または早期にフィードバックが欲しい時点で Pull Request を作成する。
6. CI（ビルド・静的解析・テスト）とレビューを通過したら `main` へマージする。
7. マージ後、作業ブランチは削除する。
8. `main` へのマージ後は速やかに（あるいは自動で）デプロイする。

### コミットメッセージ

[Conventional Commits](https://www.conventionalcommits.org/) 形式を推奨する。

    feat: 対象登録APIを追加
    fix: リフレッシュトークンのローテーション不具合を修正
    docs: ブランチ戦略をREADMEに追記

### Pull Request

- 変更内容、影響範囲、確認したテスト結果を記載する。
- 対象外の範囲へのリファクタリングを含めない（1PR = 1関心事）。
- レビュー承認後にマージする。マージ方式は Squash merge を基本とする。

### 既存の `develop` ブランチについて

GitHub Flow では `main` 一本を運用トランクとするため、`develop` ブランチは今後の恒常的な運用対象としない。新規作業はすべて `main` から分岐する。
